# External Integrations

**Analysis Date:** 2026-03-27

## APIs & External Services

**Authentication & Identity:**

- AWS Cognito - User authentication and identity provider
  - SDK: @aws-sdk/client-cognito-identity-provider
  - Auth method: OAuth 2.0, OIDC, SAML
  - Lambda triggers: Pre-Auth, Pre-Token-Generation (custom claims injection)
  - Env vars: COGNITO_JWKS_URL, COGNITO_TENANT_CLIENT_ID, COGNITO_USER_POOL_ID

**Legacy Data Bridge:**

- SQL Server (pegii) - Legacy on-premises database for data migration
  - Connection: mssql package via tenant-scoped connection strings
  - Env var: Tenant.mssqlConnectionString (per tenant, stored in Prisma)
  - Routes: /api/v1/pegii/\* (tenant-scoped)

## Data Storage

**Databases:**

- PostgreSQL (primary)
  - Provider: Neon (serverless with connection pooling)
  - Connection: Connection pooling via Neon (required for Lambda cold starts)
  - Client: Prisma 6.0.0
  - Env vars:
    - DATABASE_URL (pooled connection string, injected from Secrets Manager)
    - DIRECT_URL (optional, direct connection for migrations)
  - Schemas: Two PostgreSQL schemas
    - `public` - tenant-facing tables (customers, moves, quotes, invoices, etc.)
    - `platform` - admin-only tables (admin_users, audit_logs, feature_flags)
  - Extensions: uuid-ossp (for UUID generation)

**File Storage:**

- Not yet integrated (stub in schema via Tenant model)
- Candidate: AWS S3 (via CloudFront CDN)

**Caching:**

- Not integrated at application level
- Candidate: Redis (for multi-instance deployments)
- Current: In-memory cache via jose for Cognito JWKS

## Authentication & Identity

**Auth Provider:**

- AWS Cognito (primary)
  - Single User Pool, two app clients (tenant + admin)
  - Hosted UI for OAuth flow (authorization code)
  - TOTP MFA required for platform admins (enforced by Pre-Auth Lambda)
  - Optional for tenant users
  - Email+password auth enabled by default; can be disabled per tenant via `cognitoAuthEnabled` flag

**SSO Configuration:**

- Multi-tenancy: Each tenant can configure own identity providers
- Supported protocols: OIDC, SAML
- Storage: TenantSsoProvider model with metadata
- Secrets: IdP credentials stored in AWS Secrets Manager (ARN stored in DB, never in API responses)

**Tenant User Roster:**

- Explicit invite list (TenantUser model)
- Statuses: PENDING (invited), ACTIVE (first login complete), DEACTIVATED (revoked)
- Email + tenant unique constraint (one user per tenant per email)
- Pre-Token Lambda enforces roster (no JIT provisioning)

**API Key Auth (M2M):**

- Machine-to-machine via ApiClient model
- Key format: `vnd_<48 hex chars>` (52 chars total)
- Storage: Only SHA-256 hash persisted (plaintext returned once on create/rotate)
- Lookup: Indexed prefix search on first 12 chars
- Verification: Timing-safe equality comparison
- Scopes: Permission-based (e.g., "orders:read", "invoices:write")
- Middleware: `api-client-auth.ts` verifies hash and injects ApiClientEnv context

**Session Management:**

- Cognito-managed JWT tokens (no explicit session table)
- Tenant selection: AuthSession table (10-minute expiry) for multi-tenant picker
- Correlation ID: x-correlation-id header (UUID per request, injected by correlation middleware)

## Monitoring & Observability

**Logging:**

- @aws-lambda-powertools/logger v2 - Structured JSON logging
  - Singleton instance at `packages/api/src/lib/logger.ts`
  - Correlation ID appended to every log line via middleware
  - Log level: INFO in production, configurable via LOG_LEVEL env var
  - Output: CloudWatch Logs (retention: 1 month in production)

**Error Handling:**

- DomainError (custom error class) - Business logic violations
  - Logged at WARN level
  - Sanitized response to client (no stack traces)
- Unexpected errors - Infrastructure/runtime failures
  - Logged at ERROR level with full stack trace and correlationId
  - Sanitized response to client with code + message only

**Traces & Correlation:**

- Request correlation ID (UUID) generated by correlation middleware
- Propagated via x-correlation-id header (frontend to backend)
- Appended to all structured logs for end-to-end traceability
- Frontend injection: apiFetch (web) and adminFetch/adminFetchPaginated (admin) generate per-request UUIDs

**Alarms & Monitoring:**

- AWS CloudWatch - CloudFormation-managed alarms (in MonitoringStack)
- Metrics: Lambda invocations, errors, duration; API Gateway request count, errors
- Dashboards: Created by MonitoringStack CDK

**Audit Trail:**

- AuditLog (platform schema) - Append-only records of admin actions
  - Captured: Admin user (cognitoSub), email, action, resource type/ID, before/after JSON snapshots
  - Actions: CREATE_TENANT, UPDATE_TENANT, SUSPEND_TENANT, REACTIVATE_TENANT, OFFBOARD_TENANT
  - Immutable once written (no updatedAt)

## CI/CD & Deployment

**Hosting:**

- AWS Lambda - Serverless API (Node.js 20.x runtime)
  - Entry point: `packages/api/src/lambda.ts` (AWS Lambda adapter via Hono)
  - Cold start: Validates environment at module initialization
  - Logs: CloudWatch Logs group with 1-month retention

**API Gateway:**

- AWS API Gateway v2 (HTTP API) - REST endpoint
  - Integrated with Lambda via HttpLambdaIntegration
  - CORS enabled (configured in ApiStack)
  - CloudFront distribution (CDN) sits in front for static assets only

**Frontend Hosting:**

- AWS S3 + CloudFront
  - Tenant web app: FrontendStack (Vite SPA output)
  - Admin web app: AdminFrontendStack (Vite SPA output)
  - Cache invalidation on deploy

**Database Deployment:**

- Prisma migrations (non-destructive)
- Run on first API deployment via Lambda environment setup
- Schema: `packages/api/prisma/schema.prisma`
- Migrations: Auto-generated to `packages/api/prisma/migrations/`

**Infrastructure as Code:**

- AWS CDK (TypeScript) in `packages/infra/`
- Stacks:
  - CognitoStack - User pool, app clients, Lambda triggers
  - ApiStack - Lambda, API Gateway v2, log group
  - FrontendStack - S3, CloudFront for tenant web app
  - AdminFrontendStack - S3, CloudFront for admin web app
  - MonitoringStack - CloudWatch alarms and dashboards

**Lambda Triggers:**

- Pre-Authentication - Enforce TOTP MFA for platform admins
  - Location: `packages/api/src/cognito/pre-auth.ts`
  - Deployed via CognitoStack
- Pre-Token-Generation - Inject custom claims, activate pending users
  - Location: `packages/api/src/cognito/pre-token.ts`
  - Deployed via CognitoStack
  - Claims injected: custom:tenantId, custom:role (tenant_admin / tenant_user / platform_admin)

**Deployment Script:**

- `npm run deploy` - Runs `packages/infra/deploy.sh`
  - Synths CDK stacks
  - Deploys all stacks to AWS
  - No manual infrastructure setup required

## Environment Configuration

**Required env vars (packages/api):**

- DATABASE_URL - PostgreSQL connection string (injected from Secrets Manager)
  - Always required
  - Format: postgresql://user:password@host:port/database
- COGNITO_JWKS_URL - JWKS endpoint for JWT verification
  - Required unless SKIP_AUTH=true (dev only)
  - Format: https://cognito-idp.{region}.amazonaws.com/{poolId}/.well-known/jwks.json
- COGNITO_TENANT_CLIENT_ID - Tenant app client ID for token audience validation
  - Required unless SKIP_AUTH=true
- COGNITO_USER_POOL_ID - User pool ID for provisioning admin users
  - Required unless SKIP_AUTH=true
- SKIP_AUTH - When "true", relaxes Cognito requirements (dev/test only)
  - Optional; defaults to false (production secure mode)
- DIRECT_URL - Direct PostgreSQL connection (for migrations)
  - Optional; falls back to DATABASE_URL if not set
  - Prisma-specific: required when `directUrl` is set in schema
- LOG_LEVEL - Structured logging level
  - Optional; defaults to INFO
- NODE_ENV - Environment mode
  - Set to "production" in Lambda, "test" in tests, "development" in local dev
- PORT - HTTP server port (local dev only)
  - Optional; defaults to 3001
- HOST - HTTP server bind address (local dev only)
  - Optional; defaults to localhost
- DEFAULT_TENANT_ID - Fallback tenant for SKIP_AUTH mode
  - Optional; used in E2E tests only

**Secrets location:**

- AWS Secrets Manager
  - `pegasus/dev/database-url` - Neon pooled connection string
  - Accessed by ApiStack via CloudFormation dynamic references
  - Other secrets (SSO IdP credentials) stored per TenantSsoProvider with ARN references

**Runtime overrides:**

- Prisma engine binary path (Lambda-specific):
  - PRISMA_QUERY_ENGINE_LIBRARY=/var/task/libquery_engine-rhel-openssl-3.0.x.so.node

## API Standards

**Request/Response Envelope:**

- Success (single): `{ data: T }`
- Success (paginated): `{ data: T[], meta: { total, count, limit, offset } }`
- Error: `{ error: string, code: string }`
- HTTP status: 2xx on success, 4xx on client error, 5xx on server error

**Pagination:**

- Query params: limit, offset
- Default: limit=20, offset=0
- Meta fields: total (full count), count (this page), limit, offset

**Headers:**

- Content-Type: application/json
- x-correlation-id: UUID (injected by correlation middleware, forwarded by frontends)
- Authorization: Bearer {JWT} (Cognito tokens or API keys)

**Validation:**

- All request bodies validated against Zod schemas
- Invalid input: 422 Unprocessable Entity with detailed error messages
- Authentication: 401 Unauthorized (missing/invalid token)
- Authorization: 403 Forbidden (token valid but insufficient permissions)
